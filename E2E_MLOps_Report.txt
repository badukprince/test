Chess Analyzer AI Backend + DevOps End-to-End 구현 보고서

1) 프로젝트 주제
- 주제: 체스판 이미지 또는 FEN 입력을 기반으로 보드 상태를 추출/분석하고 유불리를 평가하는 AI 백엔드
- 목표: 완벽한 비전 인식이 아닌 현실적 구조(근사 이미지 처리 + FEN 보정 + 경량 평가 엔진)
- 핵심 가치: AI 앱 설계 + 운영 가능한 CI/CD + 컨테이너 기반 배포 자동화

2) 아키텍처 개요
- Backend: FastAPI
- Chess Logic: python-chess
- Image Heuristic: OpenCV
- Schema/Validation: Pydantic
- Container: Docker (multi-stage build)
- Automation: GitHub Actions (self-hosted runner)
- Registry/Delivery: Docker Hub

3) 구현 범위(기능)
3-1. 이미지 분석 API
- Endpoint: POST /analyze/image
- 기능:
  - 이미지 8x8 분할
  - 칸 내부 ROI 기준 점유 칸 휴리스틱 검출
  - 점유 칸을 근사 FEN으로 변환
  - 평가 엔진 호출 및 결과 반환

3-2. FEN 분석 API
- Endpoint: POST /analyze/fen
- 기능:
  - FEN 파싱 및 유효성 검증
  - 기물 점수(material) 계산
  - 중앙 장악(center control) 보너스
  - 위치 점수(positional: center/near-center occupancy) 보너스
  - 체크/체크메이트 감지 및 가중치 반영

3-3. 통합 분석 API
- Endpoint: POST /analyze
- 입력: image(optional), fen(optional)
- 동작:
  - image가 있으면 보드 추출 시도
  - 필요 시 fen 우선/보정 사용
  - 최종 평가 결과 리턴

4) 응답 모델 고도화
- score, advantage, reasoning, reasoning_ko
- total_pieces, white_pieces, black_pieces
- is_check, is_checkmate, check_side
- fen, board_matrix, source

5) 프론트 UI 구현 사항
- 단일 페이지 UI 제공(/)
- 탭 구성: 통합 분석 / FEN만 / 이미지만
- 결과 표시:
  - 점수, 우세 진영
  - 전체/백/흑 기물 수
  - 한국어 근거 + 영어 근거
  - 체크/체크메이트 상태 칩
  - 보드 매트릭스 시각화
- 안정화:
  - 로딩 무한 표시 버그 수정([hidden] 강제 우선)
  - 요청 타임아웃/전역 에러 핸들러 추가

6) 이미지 분석 개선 이력
- 초기:
  - Canny edge density 단일 기준
  - 정확도 제한(점유/색 라벨 흔들림)
- 개선:
  - 칸 내부 ROI 분석
  - 배경 추정 + 전경/윤곽/텍스처 결합 점수
  - robust threshold(MAD/Otsu 계열)
  - 점유 기물 밝기 클러스터링 기반 색 라벨
  - 상/하 보드 영역 보정 로직
- 결과:
  - 시작 배치류 이미지에서 점유 개수와 흑/백 분리 안정성 향상
  - 완전 정확 인식은 범위 밖(heuristic 한계), FEN 보정 경로 유지

7) 평가 엔진(경량 결정론) 상세
- material score: pawn=1, knight=3, bishop=3, rook=5, queen=9
- center control bonus: 중앙 4칸 공격자 수 차이 반영
- positional bonus: 중앙/준중앙 점유 보너스 반영
- tactical state:
  - check 감지 시 보정
  - checkmate 감지 시 큰 가중치 반영

8) DevOps / CI-CD 자동화
8-1. Docker
- Dockerfile: python:3.11-slim 기반 multi-stage
- 이미지 경량화/재현성 확보
- 실행 커맨드: uvicorn main:app --host 0.0.0.0 --port 8000

8-2. GitHub Actions
- Workflow: .github/workflows/ci.yml
- 트리거:
  - pull_request(main), push(main/develop), version tag(v*)
- CI 단계:
  - 의존성 설치
  - compileall sanity check
  - API smoke test(/health, /analyze/fen)
  - Docker build check
- CD 단계:
  - Docker Hub login
  - build-push-action으로 이미지 push
- Secret 적용:
  - DOCKER_USERNAME
  - DOCKER_PASSWORD

8-3. Self-hosted Runner 운영 이슈 및 해결
- 이슈:
  - setup-python 심볼릭 링크 권한 오류(Windows)
- 해결:
  - setup-python 제거
  - runner 로컬 python 재사용
  - Node20 deprecation 경고 대응(env 설정으로 Node24 사용)
- 검증:
  - Runner에서 CI Verify 성공
  - Docker Publish까지 순차 실행 확인

8-4. CI/CD 변경/수정사항 상세 이력 (확인 가능한 내역 중심)
- [변경 1] 초기 워크플로 생성
  - 상태:
    - 모든 push 브랜치에서 Docker build/push 실행하는 단일 잡 구조
  - 문제:
    - PR 검증과 실제 배포가 분리되지 않아, 변경 검증 전에 배포가 시도될 수 있음
  - 조치:
    - CI와 CD를 단계적으로 분리하는 구조로 리팩터링
  - 확인 포인트:
    - workflow 파일: `.github/workflows/ci.yml`
    - Actions 실행 시 job 분리 여부 확인

- [변경 2] CI(검증)와 CD(배포) 분리
  - 상태:
    - `ci-verify` + `docker-publish` 2잡 구조로 재구성
  - CI 내용:
    - dependency install
    - compileall sanity
    - API smoke test(`/health`, `/analyze/fen`)
    - docker build check
  - CD 내용:
    - Docker Hub login
    - metadata/tag 생성
    - 이미지 push
  - 조건:
    - `docker-publish`는 `ci-verify` 성공 후에만 실행
  - 확인 포인트:
    - Actions 화면에서 `needs: ci-verify` 의존 관계 확인

- [변경 3] 트리거 정책 정교화
  - 이전:
    - `push` 전체 브랜치
  - 변경:
    - `pull_request` to `main`: CI 검증
    - `push` to `main`/`develop`: CI 검증
    - `push` to `main` 또는 `v*` 태그: CD 배포
  - 효과:
    - 검증 단계와 배포 단계가 릴리즈 정책에 맞게 작동
  - 확인 포인트:
    - 실행 이벤트(event), ref(branch/tag) 기준 job 실행 여부

- [변경 4] Secret 키 네이밍 정렬
  - 이슈:
    - 저장소에 등록한 시크릿 이름과 워크플로 참조 키 불일치
  - 변경:
    - `DOCKERHUB_USERNAME`/`DOCKERHUB_TOKEN` -> `DOCKER_USERNAME`/`DOCKER_PASSWORD`
  - 효과:
    - Docker Hub login 실패 방지
  - 확인 포인트:
    - 실패 시 에러: authentication/credentials 관련 메시지
    - 성공 시 Docker login step pass

- [변경 5] self-hosted runner 라벨 적용
  - 변경:
    - `runs-on: [self-hosted, Windows, X64]`
  - 목적:
    - 로컬/사내 runner 인프라에서 잡 수행
  - 확인 포인트:
    - Actions job 상세의 runner name/labels
    - 예: runner `DESKTOP-GQR6GT7`, labels `self-hosted, Windows, X64`

- [변경 6] Windows runner 호환성 수정
  - 이슈:
    - Linux heredoc 기반 스크립트 문법이 Windows PowerShell 환경에서 취약
  - 변경:
    - smoke test를 `python -c` 1라인 검증 방식으로 교체
  - 효과:
    - OS 종속성 감소, self-hosted Windows 실행 안정화
  - 확인 포인트:
    - CI Verify 내 smoke step pass

- [변경 7] setup-python 권한 에러 해결
  - 증상(실제):
    - `New-Item ... SymbolicLinkElevationRequired`
    - `Administrator privilege required for this operation`
  - 원인:
    - self-hosted Windows runner에서 심볼릭 링크 생성 권한 부족
  - 변경:
    - `actions/setup-python` 제거
    - runner에 설치된 로컬 Python 사용(`python --version` 확인)
  - 효과:
    - 권한 오류 제거, CI 재실행 성공
  - 확인 포인트:
    - 동일 커밋 재실행 시 setup 단계 실패 재발 여부

- [변경 8] Node 런타임 경고 대응
  - 증상:
    - Node.js 20 deprecation warning
  - 변경:
    - workflow env에 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true` 설정
  - 효과:
    - 향후 runner 기본 Node 전환 시 호환성 리스크 완화
  - 확인 포인트:
    - Actions 로그에서 관련 경고 감소/정상 실행 확인

- [변경 9] 배포 후 운영 확인 루프 확립
  - 배포 후 확인 절차:
    - `docker pull <DOCKER_USERNAME>/chess-analyzer-api:latest`
    - `docker run --rm -p 8000:8000 ...`
    - `/health`, `/docs`, `/` 확인
  - 효과:
    - CI 성공 = 운영 동작 보장으로 연결되는 운영 체크 루프 완성
  - 확인 포인트:
    - 이미지 태그 최신 여부
    - 컨테이너 구동 및 API/UI 응답

9) End-to-End 흐름(업데이트 -> 자동 빌드 -> 자동 배포 -> 결과 확인)
- Step A: 코드 업데이트
  - 기능 개선/버그 수정/평가 로직 고도화
- Step B: Git push
  - main push 시 workflow 자동 시작
- Step C: 자동 빌드/검증(CI)
  - Python 검증 + API smoke + Docker build check
- Step D: 자동 배포(CD)
  - Docker Hub에 latest/sha/tag 이미지 업로드
- Step E: 결과 확인
  - Actions 탭 성공 여부 확인
  - docker pull 후 컨테이너 실행
  - /health, /docs, UI(/)에서 기능 동작 확인

10) 실행/검증 명령 예시
- 로컬 실행:
  - python -m uvicorn main:app --reload
- 컨테이너 실행:
  - docker pull <DOCKER_USERNAME>/chess-analyzer-api:latest
  - docker run --rm -p 8000:8000 <DOCKER_USERNAME>/chess-analyzer-api:latest
- 확인 URL:
  - http://127.0.0.1:8000/
  - http://127.0.0.1:8000/docs
  - http://127.0.0.1:8000/health

11) 최종 결론
- 본 프로젝트는 AI 앱(이미지+FEN 분석)과 DevOps 자동화(CI/CD, Docker, 배포)를 하나의 흐름으로 통합해 end-to-end 구현 완료.
- 이미지 인식은 의도적으로 heuristic 기반으로 유지하면서, FEN 보정 경로를 통해 실무형 안정성을 확보.
- 코드 변경 후 push만으로 빌드/검증/배포가 이어지는 운영 루프를 구축해, 반복 개선 가능한 MLOps 형태를 달성함.

12) 기능 업데이트 타임라인 (핵심 개선 흐름)

12-1. 초기 프로젝트 기획 및 1차 구현
- 목표:
  - 완벽 인식보다 실용적 구조(이미지 근사 + FEN 보정 + 경량 평가)
- 구현:
  - FastAPI 백엔드 + `/analyze/image`, `/analyze/fen`, `/analyze`
  - OpenCV 휴리스틱으로 8x8 분할 및 점유 칸 추정
  - python-chess 기반 기물 점수/체크 상태 평가
  - Dockerfile + GitHub Actions CI/CD 기본 파이프라인 구축

12-2. 체스 말 인식(점유 칸) 정확도 개선
- 문제:
  - 초기엔 이미지에서 기물 개수/점유 칸이 자주 누락되거나 흔들림
- 수정:
  - 칸 내부 ROI 분석으로 경계 노이즈 제거
  - 배경 추정 + 전경/윤곽/텍스처 결합 점수로 판별
  - robust threshold(MAD/Otsu 계열) 적용
  - 기물 색 라벨(백/흑)을 전역 밝기 클러스터링으로 보정
- 결과:
  - 시작 배치류 이미지에서 점유 칸 수 및 흑/백 분리 안정성 개선
  - 완전 인식은 범위 밖으로 두고 FEN 보정 경로 유지

12-3. 프론트 결과 가시성 개선
- 개선:
  - 결과 패널에 전체/백/흑 기물 수 표시
  - 한국어 근거(`reasoning_ko`)와 영어 근거 동시 표시
  - 로딩 무한표시 이슈 수정([hidden] 우선 적용)
  - 요청 타임아웃/예외 처리 보강
- 효과:
  - 분석 결과 해석성과 사용자 피드백 품질 향상

12-4. 중앙 컨트롤 + 위치 점수(포지셔널) 강화
- 초기:
  - 중앙 4칸 공격자 수 차이만 반영
- 수정:
  - 중심 4칸 + 준중심 12칸 점유 기반 위치 점수 항목 추가
  - reasoning에 `positional bonus (center occupancy)` 노출
  - 한국어 번역 항목 동기화
- 효과:
  - 단순 material 외에 위치적 우세를 경량 방식으로 반영

12-5. 체크/체크메이트 상태 감지 및 UI 연동
- 수정:
  - API 응답에 `is_check`, `is_checkmate`, `check_side` 필드 추가
  - FEN 평가 함수에서 체크/메이트 상태를 구조화해 반환
  - 프론트에 전술 상태 칩 추가(정상 / 체크 / 체크메이트)
- 효과:
  - 전술 위험 상태를 점수 외에 명시적으로 확인 가능

12-6. 최종 운영 흐름 정착
- 개발 루프:
  - 기능 업데이트 -> push -> 자동 CI 검증 -> 자동 Docker 배포 -> pull/run 결과 확인
- 의미:
  - 모델 학습 없이도, 실무형 AI 백엔드 개선 사이클을 안정적으로 반복 가능한 구조 완성
